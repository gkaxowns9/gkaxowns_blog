## 객체 지향 프로그래밍의 정의

- 프로그램 설계 방법론 중 하나로, 개발 가이드라인 중 하나라고 생각하면 쉽다.
  프로그램을 여러 객체(Object)로 나누고, 객체간의 상호작용을 통해 개발하는 방법이다.
  프로그램을 여러 객체로 나누게 되면서, 동작들을 직관적으로 보여 주는 효과가 있다.

## 객체

- 객체는 프로그램에서 **여러 변수나 함수를 담고 있는 공간**이고, 보통 변수로 저장된다.
  **속성과 메소드로 구성**되며, Class를 선언하고 불러와 속성값을 부여하여 사용한다.
  객체 지향 프로그래밍 언어(Java, Python JS 등)으로 간단한 예시를 들어보자.

```python
#파이썬을 사용한 예시 코드
import random #Class를 import함

rad = random() #Class를 사용해서 객체를 생성해 rad 라는 변수에 저장함
print(rad.randint(1,10))
```

- 이와 같이 Class를 가져와서 변수에 객체를 저장하고, 객체에 담겨져 있는 변수나 함수를 불러와 사용하게 된다.
  Class와 객체의 관계에 대해 비유하자면 붕어빵 틀과 붕어빵이라고 생각하면 쉬울 듯 하다.

  Class를 선언하는 방법은 Java로 예시를 들어보자.

```java
//자바를 사용한 예시 코드
public class test {
	public int a; //속성
	private String b = "Hello, OPP!";

	public test(int a) { //생성자
		this.a = a;
	}

	public void testfunc1() { //메소드1
        System.out.println(b);
    }

    public int testfunc2() { //메소드2
        return a;
    }
}
```

### 객체의 상호작용

- 위 예시들처럼 선언된 객체들의 상호작용이 어떤 방식으로 이루어지는지 확인해보자.
  가장 많이 쓰이는 상호작용은 함수를 이용하는 방법으로, 설명하자면 객체A 안에서 다른 객체, B의 함수를 불러와 매개변수에 적절한 값을 주어 함수에 리턴하는 값을 받아오는 방식이다.
  상호작용의 방법에 대해서는 어떻게 프로그래밍하는지에 달렸지만, 대부분 위의 방법을 사용한다.
- 아래는 파이썬으로 작성한 딕셔너리, 리스트 타입의 객체의 상호작용을 나타내는 예시로, 두 사람이 서로 메시지를 주고 받는 상황을 표현한 코드이다.
  ```python
  # 파이썬을 사용한 예시 코드
  alice = {"name": "Alice", "messages": []} # alice 객체 생성
  bob = {"name": "Bob", "messages": []} # bob 객체 생성

  # 메시지 주고받기
  message = "Hi Bob, how are you?"
  bob["messages"].append(message)
  print(f"{alice['name']} sent a message to {bob['name']}: {message}")

  reply = "Hi Alice, I'm good! How about you?"
  alice["messages"].append(reply)
  print(f"{bob['name']} replied to {alice['name']}: {reply}")
  ```

## 특징

- 객체 지향 프로그래밍의 특징은 **추상화, 캡슐화, 상속, 다형성**으로 4가지로 나눌 수 있다.

### 추상화 (Abstraction)

- 객체들이 공통적으로 **필요로 하는 속성이나 동작을 하나로 추출해내는 작업**으로, 객체들의 공통적인 부분을 따로 추출하여, 하나의 묶음으로 만들어 내는 것이다.

### 캡슐화 (Encapsulation)

- 객체에 필요한 **변수와 함수를 하나의 단위로 묶는 작업**으로, 보통 Class를 통해 구현한다.
  위 Class 선언 예시에서 보여준 Class 안에서 변수, 함수들을 선언하는 것이 캡술화이다.
  여기서 추가로 캡슐화된 변수, 함수들이 다른 객체로 드러나지 않게 감출 수 있는데, 이것을 **은닉**이라고 한다. (JS에서는 존재하지 않는 것으로 알고, Python에서는 사용을 지양하고 있다. 쓰지말라고 이런저런 조건들을 붙혀놓았다.)

### 상속 **(Inheritance)**

- 여러 객체들이 지닌 **공통된 특성을 부각시켜 하나의 상위 객체으로 정의하는 작업**이다.
  여기에서 정의된 상위 객체를 **부모**, 공통된 특성을 가진 객체들을 **자식**이라 한다.
  자식 객체가 부모 객체의 모든 메서드을 따르지 않을 때에는 오버라이딩(Overriding)으로 그 메서드을 변경할 수 있다. (오버라이딩은 다형성의 기능이라고 보는 것이 더 적절하다고 생각하고 있기에 댜형성에서 설명하겠다.)

### 다형성(polymorphism)

- 하나의 변수, 또는 함수가 상황에 따라 다른 의미로 해석될 수 있는 것을 말한다.
  각 상황에 따라 사용되는 기능이 여러가지이고, 여기서는 2가지만 설명하겠다. - 오버라이딩(Overriding): **부모 클래스의 메서드를 자식 클래스에서 재정의해서 사용하는 것**을 말하며, 자식 클래스는 부모 클래스의 메서드를 재활용하면서, 독자적인 기능을 추가할 수 있다.
  부모 클래스 대신 내용이 없는 메서드를 가지는 **인터페이스(Interface)**를 상속하여 메서드를 무조건 오버라이딩하게 만들 수도 있다. - 오버로딩(Overloading): **같은 이름의 메서드를 다양한 매개변수 타입과 개수에 따라 다르게 선언하여 사용하는 것**을 말하며, 오버로딩은 메서드 이름을 동일하게 유지하면서도, 다양한 상황에서 유연하게 대응할 수 있는 방법을 제공한다.

## SOLID 원칙

- 객체 지향 프로그래밍에서 가장 중요한 설계 원칙 5개의 약자이다.
  단일 책임 원칙 (**S**RP), 개방-폐쇄 원칙 (**O**CP), 리스코프 치환 원칙 (**L**SP), 인터페이스 분리 원칙 (**I**SP), 의존 역전 원칙 (**D**IP) 와 같이 총 5개의 설계 원칙을 부르는 용어이다.

### 단일 책임 원칙 (**S**RP)

- **하나의 클래스에서는 하나의 책임을 저야한다**는 원칙으로, 클래스 하나가 다른 분야의 작업을 하면 안 된다는 내용이다.
  간단하게 예를 들자면, `Login` 클래스에서는 유저의 로그인과 관련된 동작만 수행해야 하고, 상품 구매, 알림 등과 같은 다른 분야와 관련된 동작을 하면 안 된다는 소리이다.

### 개방-폐쇄 원칙 (**O**CP)

- **기존 코드를 수정하지 않고 새로운 기능을 추가할 수 있어야 한다**는 원칙으로, 프로젝트 구조가 확장은 쉽고 (확장 개방적), 변경은 어려워야 한다는(변경 폐쇠적) 내용이다. 위에서 설명한 4개의 특징 중 다형성의 요소들을 활용하여, 기존 코드를 최대한 변경하지 않고, 새 기능 확장을 할 수 있어야 한다는 소리이다자식 클래스는 언제나 부모 클래스를 대체할 수 있어야 한다.

### 리스코프 치환 원칙 (**L**SP)

- **자식 클래스는 언제나 부모 클래스를 대체할 수 있어야 한다**는 원칙으로, 자식 클래스가 부모 클래스의 메서드를 오버라이딩 했을 때, 완전히 다른 동작을 하면 안 된다는 내용이다.
  다른 말로는, 부모 클래스에서는 문제없는 메서드가, 자식 클래스에서 예외를 던지면 안 된다는 내용이다.

### 인터페이스 분리 원칙 (**I**SP)

- **클라이언트가 사용하지 않는 인터페이스에 의존하지 않아야 한다**는 원칙으로, 인터페이스를 상속받는 클래스가, 인터페이스가 요구하는 메서드가 필요없다면, 해당 인터페이스를 상속하면 안 된다는 내용이다.
  간단하게 예를 들자면, `Worker` 인터페이스가 `eat` 메서드를 요구할 때, 그를 상속받는 `Robot` 클래스가 `eat` 메서드가 필요 없으므로, `Worker` 인터페이스를 상속하면 안 된다는 소리이다.

### 의존 역전 원칙 (**D**IP)

- **상위 모듈이 하위 모듈에 의존하지 않고, 추상화(인터페이스)에 의존해야 한다**는 원칙으로, 하위 모듈에 직접 의존하면 코드의 유연성이 떨어지기에 안 된다는 내용이다.
  간단하게 예를 들자면, `Computer` 클래스가 `Keyboard` 인터페이스를 상속받는 하위 모듈인 `Logitech` 클래스에 직접 의존하면, `Computer` 클래스는 다른 키보드를 사용하지 못하고 오직 로지텍 키보드만 사용할 수 있게 되어 안 된다는 소리이다.
- 객체 지향 프로그래밍 설계를 하면서 발생하는 **반복적인 문제들을 해결하기 위한 방안**이다.
  많은 개발자들이 프로젝트 구조를 구성하며 실제 산업에서 겪은, 겪을 어려움을 해결하기 위해 만들었기에 **재사용성, 가독성, 확장성, 유지보수성, 안정성 등을 보장**할 수 있다.

## **디자인 패턴**

- 이러한 디자인 패턴은 크게 **생성(Creational)**, **구조(Structural)**, 그리고 **행위(Behavioral)** 패턴으로 분류된다.

### 생성 패턴

- 생성 패턴은 객체 생성 방식을 다루며, 객체 생성 과정을 캡슐화하여 코드의 유연성과 재사용성을 높이는데 중점을 둔 패턴들을 말한다. 다양한 패턴들이 존재하지만 여기서는 3개만 설명하겠다.
  - 싱글톤(Singleton) 패턴: **특정 클래스의 인스턴스가 오직 하나만 생성되도록 보장**하는 디자인 패턴이다. 인스턴스가 프로그램 전체에서 동일한 자원을 공유하도록 하는데 유용하다.
    예를 들어, 애플리케이션에서 설정 정보를 관리하는 클래스는 싱글톤 패턴을 사용하여 하나의 인스턴스만 존재하도록 할 수 있다.
  - 팩토리 메서드(Factory Method) 패턴: **객체 생성을 위한 인터페이스를 정의하여 초기화 과정을 외부에서 보지 못하게 숨기고 반환 타입을 제어**하는 디자인 패턴이다. Factory(공장)이라는 이름처럼, 객체를 생성하는 인터페이스, 공장을 하나 만들고 그 공장에 다양한 객체를 생성할 수 있는 하위 클래스를 객체별로 각각 만들어서 클라이언트 코드와 객체 생성 로직을 분리하여 코드의 유연성을 높일 수 있다.
    (인터페이스가 공장이고, 하위 클래스가 공장안의 객체 생성 장치같은 느낌?)
  - 빌더(Builder) 패턴: **인스턴스를 생성자를 통해 직접 생성하지 않고, 빌더라는 내부 클래스를 통해 간접적으로 생성**하는 디자인 패턴이다. 메서드 체이닝을 지원하여 복잡한 객체의 생성 과정을 단계별로 분리하여 객체를 만들 수 있다.
    예를 들어, 여러 옵션들이 필요한 JSON 생성의 경우, 빌더 패턴을 사용하여 단계적인 작업으로 나누어 객체를 생성할 수 있다.

### 구조 패턴

- 구조 패턴은 클래스나 객체를 조합하여 더 큰 구조를 형성하며, 인터페이스를 단순화하거나 객체 간의 관계를 효율적으로 구성하는 데 중점을 둔 패턴들을 말한다. 마찬가지로 3개만 설명하겠다.
  - 데코레이터(Decorator) 패턴: **데코레이터를 사용하여 객체의 메서드에 기능을 추가**하는 디자인 패턴이다. 기본 기능에 추가적인 기능을 덧붙이는 방식으로, 객체에 동적으로 새로운 기능을 추가하여 유연하게 확장할 수 있다.
  - 어답터(Adapter) 패턴: **호환되지 않는 인터페이스를 가진 클래스들을 함께 사용할 수 있도록 중간에 어답터를 두어 인터페이스를 변환**하는 디자인 패턴이다.
  예를 들어, 기존 코드와 아얘 다른 라이브러리나 파일 구조를 가진 신 기능 코드를 도입할 때 어답터 패턴을 사용하여 두 기능이 같은 형식 아래 동작할 수 있게 한다.
  - 파사드(Facade) 패턴: **내부의 복잡한 처리들을 대신 수행해서 결과만 넘겨주는 객체를 만드**는 디자인 패턴이다. 참고로 facade는 프랑스어로 외벽을 뜻하는 말이라고 하며 내부의 복잡한 처리들을 숨겨주는 외벽을 만든다고 생각하면 쉽다.

### 행위 패턴

- 행위 패턴은 객체나 클래스 사이의 상호작용과 책임 분배에 중점을 두는 디자인 패턴들을 말한다.
  객체가 가져야 할 메서드를 분배하고 알고리즘이나 흐름 제어를 관리한다. 2개만 설명하겠다.
  - 옵저버(Observer) 패턴: **한 객체의 상태 변화가 있을 때, 그 객체에 의존하는 다른 객체들에게 자동으로 통지하여 상태를 갱신할 수 있도록** 하는 디자인 패턴이다. 주로 이벤트 처리 시스템에서 많이 사용된다. 
  - 전략(Strategy) 패턴: **여러 알고리즘을 정의하고, 런타임 시에 이들 중 하나를 선택하여 사용할 수 있도록** 하는 디자인 패턴이다. 알고리즘을 클라이언트로부터 분리하여 독립적으로 변경하거나 확장할 때 전략 패턴을 사용한다.
